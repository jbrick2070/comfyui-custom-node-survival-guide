# Bonus — three-version YAML normalization experiment

This folder is a **bonus** for readers curious about how different AI models
handle the same "clean up this YAML" task. Nothing in here is authoritative
content, and none of it ships in the bible itself — these are scaffolding
files for an experiment you can run on your own machine.

## What's here

| File | Role |
|------|------|
| `BUG_BIBLE.v0_baseline.yaml`         | Snapshot of `BUG_BIBLE.yaml` BEFORE any normalization pass |
| `BUG_BIBLE.v1_claude.yaml`           | Snapshot AFTER the Claude (Opus class) pass — same as the live bible at the time of commit |
| `round_robin_question.md`            | The question fed to the round-robin addon |
| `run_pass_b.ps1`                     | PowerShell block that calls the addon to produce v2 |
| `compare_passes.py`                  | Diffs v0 / v1 / v2 and prints + saves a comparison report |
| `compare_report.md`                  | Generated by `compare_passes.py` |

## Why bother

Two reasons:

1. **It's a real test of the `llm_round_robin/` addon.** Run it on something
   non-trivial (~3000 lines of YAML, 153 entries) and see whether the
   round-robin synthesis surfaces useful normalizations the first pass missed.
2. **Comparing two AI normalization passes is a useful exercise on its own.**
   Where do the two passes agree? Where do they disagree? Are the
   disagreements substantive or stylistic? Either answer teaches something
   about how to budget AI-assisted refactors on a knowledge base you maintain.

## How to run

You need at least one of `OPENAI_API_KEY` / `GEMINI_API_KEY` / `NVIDIA_API_KEY`
set as a User env var.

```powershell
cd /d C:\Users\you\Documents\ComfyUI\comfyui-custom-node-survival-guide
.\docs\bonus_normalization_experiment\run_pass_b.ps1
```

That writes per-provider responses + a synthesis markdown into the same
folder. Read the synthesis. Pick the recommendations you want to apply
(or take the union, or take none — the synthesis ends with a checklist
prompting you to decide). Save the post-application file as
`BUG_BIBLE.v2_round_robin.yaml` next to v0/v1.

Then:

```powershell
python docs\bonus_normalization_experiment\compare_passes.py
```

That prints + saves `compare_report.md`: line-count delta, schema status of
each version, count of common / Claude-only / round-robin-only changes, and
samples of each.

## What I'd watch for

- **Does the round-robin's synthesis catch something Claude's pass missed?**
  The Claude pass focused on mechanical normalizations (typo fixes, tag
  dedup, legacy_id quoting, whitespace). The round-robin question is shaped
  to find structural issues (phase mis-classification, OTR-narrative leaks,
  missing cross-links, coverage gaps).
- **Do the three providers agree?** If all three converge, the recommendation
  is probably solid. If two-vs-one, the synthesis flags it. If they
  three-way-disagree, the recommendation isn't ready to apply.
- **Does the round-robin stay grounded?** The question deliberately doesn't
  paste the whole 3000-line YAML — the providers reason from the schema
  summary + their training knowledge. Hallucinated entry IDs are a smell.

This is a bonus, not a recommendation. If you ship YAML you maintain, you
own the call about what normalizations to apply. The experiment just lets
you compare what different AI normalizers would suggest.
